# Manual QA Evidence

Use this after `npm run qa:release:auto` and before publishing a release. The JSON evidence file lives under ignored `release-artifacts/` because it is release-specific. Do not rerun automated QA after filling the manual evidence unless you update `automatedEvidenceGeneratedAt` to match the new automated evidence.

1. Generate a template:

```sh
npm run qa:release:manual:template
npm run qa:release:manual:profile -- --sync-template --launch
```

When `release-artifacts/release-qa-automated.json` exists and still matches the current worktree, build fingerprint, bundled Whisper fingerprint, command inventory, versions, release manifest surface, and passed built-extension browser evidence, the template pre-fills `automatedEvidenceGeneratedAt`, `environment.extensionSource`, and the unpacked extension ID actually observed by Chrome. Stale or incomplete automated evidence is ignored during template generation, leaving placeholders in place so the releaser reruns automated QA first. The template requires a tester name for attribution but does not request an email address or other contact data; intentional legacy contact fields remain optional. The template starts with `status: "template"`; set it to `status: "passed"` only after completing the real checklist with release-specific observations. Required confirmation booleans intentionally start as `false`, and checklist entries intentionally start as `status: "template"`; flip them only after observing the corresponding manual result. The template writer refuses to overwrite an existing `release-artifacts/manual-qa-evidence.json`; use `npm run qa:release:manual:template:force` only when intentionally replacing an unsubmitted template. The automated evidence path, build path, bundled Whisper path, and manual `environment.extensionSource` must remain canonical relative paths such as `release-artifacts/release-qa-automated.json`, `build`, and `build/assets/whisper`; absolute machine-local paths are rejected. `testedAt` and all observation fields still need to be filled after the real manual QA pass.

The template also names the canonical media and sidecar probe reports and includes filename, byte-size, and SHA-256 fields for source recordings, exports, and project-audio inputs. Alternate or absolute report paths are rejected.

`npm run qa:release:manual:profile -- --sync-template` fails closed unless `release-artifacts/release-qa-automated.json` has `status: "passed"`, fingerprints both the current canonical `build/` and current git worktree, and carries a valid extension ID observed by the clean-profile packaged-extension smoke. It additively merges fields missing from the canonical current template, including nested environment fields, removes recognized retired zoom fields and the old tester-email placeholder only when they still equal their untouched template values, then updates release version, automated-evidence path/timestamp, and clean-profile build fields on an existing `status: "template"` manual evidence file. It detects the host OS and selected Chrome version and carries forward the browser-observed extension ID, prefilling only untouched canonical placeholders; tester-entered values are never replaced, and unavailable browser-version detection remains manual. Verify all prefilled values during the session. User-entered legacy values and unknown fields remain intact. Synchronization converges without further changes when repeated against the same template/evidence and refuses to modify non-template evidence. After that preflight, it prints a clean-profile Chrome command for the current `build/` and a temporary user-data directory. Its `--json` output includes `templateSynchronized`, `browserObservedExtensionId`, `detectedEnvironment`, and `evidencePrefill`. If you pass `--profile-dir=...` after the npm `--` separator, the directory must be new or empty. The helper opens `chrome://extensions/` so the prefilled ID can be visually confirmed; add `--launch` after the npm `--` separator to start Chrome directly, or set `SAYLESS_CHROME` if Chrome is not in a standard location.

The helper prepares each new directory with a `.sayless-manual-qa-profile.json` marker bound to the release version, automated-evidence timestamp, build hash/count/size, browser-observed extension ID, detected OS, and selected browser executable/version. Its output includes a copyable resume command. If Chrome closes before the long session is finished, use that exact command, or run `npm run qa:release:manual:profile -- --profile-dir=/the/printed/path --resume-profile --launch`. Resume succeeds only for a marked session whose provenance still matches the current passing automated evidence and test environment. Unmarked existing Chrome profiles, malformed markers, changed builds, rerun automated evidence, OS/browser changes, and `--resume-profile` combined with template synchronization fail closed.

A successful `--launch` waits for the operating system to acknowledge the browser process, then atomically stamps `manualSession` in `release-artifacts/manual-qa-evidence.json` and records the active profile in ignored, machine-local `release-artifacts/manual-qa-session.json`. The portable stamp binds the profile creation time, release and automated-evidence timestamps, build hash/count/size, extension ID, OS, and browser version; it deliberately omits the profile directory and browser executable path and makes no claim about tester-owned observations. A launch against stale evidence fails closed. Repeating template synchronization for the same automated run preserves the stamp, while synchronization to newer automated evidence resets it to placeholders so a fresh launch is required.

Run `npm run qa:release:status` after synchronization or when returning to the checklist. It compares the template with the canonical generated schema and current automated provenance: a stale or incomplete template advances to `npm run qa:release:manual:profile -- --sync-template --launch`; a current template advances to the exact recorded resume command only after the profile helper strictly revalidates the pointer and marker, otherwise it advances to a new `npm run qa:release:manual:profile -- --launch`. The JSON report exposes `manualQa.templateSyncRequired`, `manualQa.templateSyncReasons`, and `manualQa.activeSession`. The machine-local session pointer is not release evidence and must not be attached; the path-free `manualSession` object inside the manual evidence is.

Both commands delegate merge, retired-placeholder migration, provenance synchronization, and freshness analysis to `scripts/manual-qa-template-sync.mjs`. The release audit prevents those rules from being copied back into either caller, keeping the read-only recommendation aligned with the write operation.

2. Fill in real test results from `docs/RELEASE_QA.md`, using
   `npm run qa:release:manual:progress` throughout the clean-profile session.
   Keep every downloaded source, export, project-audio input, and sidecar. Run the
   strict probes below only after those inputs exist; their dedicated progress
   sections deliberately follow the human recording/export/checklist sections.

Measure downloaded source recordings, media exports, and WAV/M4A/MP3 project-audio inputs instead of transcribing file properties by hand:

```sh
npm run qa:release:manual:media -- --json --require-complete --output=release-artifacts/manual-qa-media-probe.json /path/to/source.mp4 /path/to/source.webm
npm run qa:release:manual:media -- /path/to/input.wav /path/to/input.m4a /path/to/input.mp3
```

The read-only probe streams each file through the same media parser used by the product and reports its exact byte count, SHA-256, detected format/MIME type, duration, video display dimensions/codec, and audio channels/sample rate/codec. JSON includes schema-exact `recordingFields` for MP4/WebM sources and `projectAudioInputFields` for WAV/M4A/MP3 inputs, per-video 180-second/25-MiB threshold results, and an informational `releaseCoverage` summary for source count, MP4/WebM containers, one combined long-and-large candidate, varied dimensions/aspect ratios, and all three project-audio formats. `--require-complete` retains the JSON/human report but exits nonzero until all five measurable checks pass; omit it while incrementally collecting files. `--output=<report.json>` atomically writes the same JSON report, including on an incomplete strict run, and refuses to overwrite an inspected input. Keep the recommended ignored `release-artifacts/manual-qa-media-probe.json` with the session evidence. Because exported video cannot be distinguished from an original source by container metadata, confirm the source role manually. The helper does not edit the worksheet or claim that a file was played, visible, audible, synchronized, or acceptable; record those human observations separately. Unsupported, empty, missing, or non-file inputs fail closed.

The final strict invocation must include every worksheet source recording, MP4/WebM/GIF/WAV/M4A export, and WAV/M4A/MP3 project-audio input. The manual gate requires complete coverage, requires the report timestamp to fall between automated QA and `testedAt`, and cross-checks filenames, byte sizes, hashes, durations, dimensions, containers, and project-audio decode metadata. It still cannot establish source role, playback, visual quality, audibility, or synchronization.

Validate the three required non-media export structures together:

```sh
npm run qa:release:manual:sidecars -- --json --require-complete --output=release-artifacts/manual-qa-sidecar-probe.json /path/to/recording.vtt /path/to/recording.transcript.json /path/to/recording.sayless-project.json
```

The sidecar probe verifies monotonic non-empty WebVTT cues, the SayLess transcript kind/schema plus positive timed source and timeline-aware words, and the SayLess project kind/schema with current project version 4, matching recording identity, bounded unique timeline clips, matching source durations, and a supported export format. It reports exact bytes, SHA-256 identities, schema-exact `exportFields`, useful counts, and per-filename sidecar sets. `coverage.status` is `structurally-complete` only when at least one common filename stem supplies VTT, transcript JSON, and project JSON and the two JSON recording IDs agree; three formats split across recordings remain incomplete. `--require-complete` retains the report but exits nonzero when no matched set is complete; omit it while incrementally collecting files. `--output=<report.json>` atomically writes the report even when strict coverage is incomplete and refuses to overwrite an input; retain the recommended ignored `release-artifacts/manual-qa-sidecar-probe.json`. Missing file paths, malformed structure, renamed formats, substituted kinds, or old project versions fail closed. The local-recordings browser harness runs the same strict CLI mode against product-generated exports. Structural success does not prove the VTT/transcript was opened or the project was imported; those remain explicit manual observations.

The manual gate requires this canonical report to be strict and structurally complete, checks that it was generated after automated QA and before `testedAt`, and cross-checks worksheet VTT/transcript/project filenames, formats, byte sizes, and SHA-256 identities. Opening the text exports and importing the project remain human observations.

Once both canonical strict reports pass and the worksheet filenames already identify the tester-confirmed source, export, and project-audio roles, preview `npm run qa:release:manual:measurements -- --json`, then add `--write` to apply the measurement import atomically. It requires exact filename/format matches for every listed recording, export, and project-audio input and copies only probe-measured hashes, byte sizes, durations, dimensions, containers, channel counts, and sample rates. It rejects stale or incomplete reports, mismatches, missing entries, and non-template evidence. It never infers roles, changes filenames, approves checks, sets `testedAt`, or supplies playback, visual, audibility, synchronization, reveal, open, import, or other human observations; repeating it against unchanged inputs makes no further changes.

During the session, review structured progress without weakening the release gate:

```sh
npm run qa:release:manual:progress
npm run qa:release:manual:progress -- --json
npm run qa:release:manual:progress -- --section=recordings
```

Progress mode runs the authoritative validation rules, groups remaining issues into manual QA sections, and shows up to three sample issues per section. It names the first incomplete section and prints the exact focused command to continue with it. Pass `--section=<section-id>` to show every remaining verifier issue for one section while retaining the overall counts; JSON output exposes the same `nextSection` and `selectedSection` data for tooling. Probe-report failures have dedicated `mediaProbe` and `sidecarProbe` sections and route to the matching strict collection command instead of incorrectly asking for another automated QA run; genuine automated-build or provenance failures still route to `npm run qa:release:auto`. Once those reports are valid, hashes, sizes, durations, dimensions, containers, channels, and sample rates that differ from the reports move into a later `measurementImport` section whose command runs the guarded importer. Filenames, formats, roles, and observations stay in their human-owned sections. The network-disabled environment confirmation is grouped with offline transcription rather than initial metadata because it is established during that test. The terminal `status: "passed"` and `testedAt` requirements are deferred to a `finalization` section after both probe reports and measurement import, so the queue does not ask the tester to approve or timestamp incomplete evidence. Unknown section ids fail with the available ids. Progress remains read-only and exits successfully for a valid JSON worksheet even while sections remain incomplete, so it is suitable for a checklist display or automation. Missing or invalid JSON still fails. A `complete` progress report is informational; only the strict verifier below can approve evidence for packaging.

Every section also exposes canonical `workTargets`. Human output prints them for the focused or next section, while JSON includes them on every section plus `nextSection` and `selectedSection`. Targets identify the worksheet object to edit—such as `recordings`, `projectAudio`, or final `status`/`testedAt`—or the generated artifact involved, without filling or claiming any observation.

The strict verifier also requires automated evidence to carry a valid browser-observed extension ID, canonical `build` path, clean temporary-profile provenance, an observation timestamp inside the automated run, and a positive surface-summary count. The manual clean-profile worksheet must report that same ID; a different valid-looking ID is rejected.

Manual evidence must also carry the machine-stamped `manualSession` object created after an acknowledged launch. Its profile creation time must fall between automated QA and `testedAt`, and its release, automated timestamp, build hash/count/size, extension ID, OS, and browser version must exactly match the corresponding automated and environment evidence.

3. Verify the evidence:

```sh
npm run qa:release:manual
```

The verifier requires `status: "passed"`, the release version to match `package.json`, `package-lock.json`, and `src/manifest.json`, the referenced automated evidence file to exist at the canonical relative path, `automatedEvidenceGeneratedAt` to match that file's `generatedAt`, `testedAt` to be at or after that automated evidence timestamp, and the automated evidence itself to carry `status: "passed"` with internally consistent run window, total duration, command durations, git branch/commit/dirty-state, and exact worktree fingerprint. Automated evidence must include each required release QA command exactly once with `status: "passed"` including `test:e2e:offline-transcription-smoke`, the expected executed command string, and a non-negative duration; unexpected or duplicate command records are rejected. The current `build/manifest.json` version, canonical relative `build/` path, `build/` fingerprint, build byte count, build formatted size, canonical relative bundled-Whisper path, bundled-Whisper fingerprint, bundled-Whisper byte count, and bundled-Whisper formatted size must match the automated evidence. Manual evidence must also confirm a clean Chrome profile, record the 32-character unpacked extension id, and point `environment.extensionSource` at the same canonical relative `build` path fingerprinted by automated QA. Package and CWS evidence preserve and verify the same automated git/worktree provenance so published ZIPs remain tied to the source state that produced the automated QA pass. It also rejects unedited template or fixture placeholder text. Finally, it requires at least two real recordings with unique ids, exact positive-integer source-file `byteSize`, `width`, and `height` values, microphone descriptions, and noise profiles that cover MP4 and WebM, tab/browser/region and desktop/screen/window capture/recording sources, at least one recording that is both 180 seconds or longer and at least 25 MiB, at least two speaker profiles, and recording notes that describe observed, confirmed, verified, recorded, inspected, tested, opened, or reviewed local/offline recording behavior. It requires inspected exports for MP4, WebM, GIF, WAV, M4A, WebVTT, transcript JSON, and `.sayless-project.json` tied back to listed recording ids with unique filenames matching the recorded export format and notes that describe how each export was opened, played, imported, decoded, previewed, viewed, loaded, listened to, or inspected, verified caption burn-in, export cancel/retry tied to that same long-and-large real recording and a video format, reveal action evidence that mentions a completed export/download and the Chrome download id, reveal, open, or show-in-folder observation, Save to file evidence that describes the user-chosen file/folder flow or explains why File System Access was unavailable, save-dialog cancellation evidence, structured offline-transcription evidence for bundled-model-ready UI status, a concrete disabled/blocked/offline network isolation method, a failed external HTTP(S) network probe URL/error from the same Chrome profile, a probe result that says the bundled/local Whisper model stayed ready or loaded, transcript quality notes that mention real-speaker/voice/recording quality with word timing, timestamps, accuracy, or usability, cache/regenerate/delete with network disabled, structured silence-suggestion evidence linked to distinct listed MP4 and WebM recordings with at least two noise profiles, an observed suggested quiet range for each referenced recording, and an observed noisy range that was not suggested, per-recording structured zoom evidence for at least two tab/browser/region sources with distinct dimension pairs and aspect ratios covering click metadata, keep/remove, preview, persisted keyframes after reopen, and inspected MP4 export framing, per-recording structured crop evidence across at least two distinct source dimension pairs and aspect ratios covering normalized edge bounds, native controls, non-destructive preview, reopen persistence, and inspected MP4 dimensions whose aspect ratio matches the crop, real WAV/M4A/MP3 project-audio inputs with decoded metadata and audible previews plus long-recording synchronization, perceptual mix/replace/loop checks, cancellation/retry tied to the long-large export recording, reopen, duplicate/delete isolation, sidecar missing-asset/relink, and Apply-edits cleanup evidence, structured local-library recovery evidence with duplicate-reopen, sidecar-import, bulk-export-delete, orphan-cleanup, and missing-media-repair operation observations, reviewed release notes, screenshots, `docs/STORE_LISTING.md`, and final store text with searched paid/subscription, premium/trial/entitlement/license/upgrade, plan/membership/locked-feature, locked-behind/pay-to-unlock/upgrade-required gates, account-tier/license-key/activation/contact-sales gates, account/sign-in, and cloud/remote terms, and a `store-text` reviewed artifact name that includes `docs/STORE_LISTING.md`. Publication notes must describe reviewing paid/account/cloud/remote/local-only claims, residual-risk fields must explicitly say no residual risk/no remaining risk or describe the residual risk, and structured checklist evidence must pass for fresh install, recording recovery, timeline persistence, and final no-paid/no-cloud surface checks. Checklist notes and observations must describe observed, confirmed, verified, recorded, inspected, tested, opened, or reviewed local/offline release behavior, and checklist artifacts must identify screenshots, reports, logs, recordings, exports, transcripts, videos, images, JSON/VTT/MP4/WebM/GIF/WAV/M4A files, or notes. Multi-recording evidence requirements must reference distinct listed recording ids; repeating the same id does not satisfy two-recording coverage. Real-recording checklist evidence must identify the listed recording ids used for that check.
