# Manual QA Evidence

Use this after `npm run qa:release:auto` and before publishing a release. The JSON evidence file lives under ignored `release-artifacts/` because it is release-specific. Do not rerun automated QA after filling the manual evidence unless you update `automatedEvidenceGeneratedAt` to match the new automated evidence.

1. Generate a template:

```sh
npm run qa:release:manual:template
npm run qa:release:manual:profile
```

When `release-artifacts/release-qa-automated.json` exists and still matches the current worktree, build fingerprint, bundled Whisper fingerprint, command inventory, versions, and release manifest surface, the template pre-fills `automatedEvidenceGeneratedAt` from that file and sets `environment.extensionSource` to the automated build path. Stale or incomplete automated evidence is ignored during template generation, leaving the timestamp placeholder in place so the releaser reruns automated QA first. The template starts with `status: "template"`; set it to `status: "passed"` only after completing the real checklist with release-specific observations. Required confirmation booleans intentionally start as `false`, and checklist entries intentionally start as `status: "template"`; flip them only after observing the corresponding manual result. The template writer refuses to overwrite an existing `release-artifacts/manual-qa-evidence.json`; use `npm run qa:release:manual:template:force` only when intentionally replacing an unsubmitted template. The automated evidence path, build path, bundled Whisper path, and manual `environment.extensionSource` must remain canonical relative paths such as `release-artifacts/release-qa-automated.json`, `build`, and `build/assets/whisper`; absolute machine-local paths are rejected. `testedAt` and all observation fields still need to be filled after the real manual QA pass.

`npm run qa:release:manual:profile` fails closed unless `release-artifacts/release-qa-automated.json` has `status: "passed"` and fingerprints both the current canonical `build/` and the current git worktree state. After that preflight, it prints a clean-profile Chrome command for the current `build/` and a temporary user-data directory. Its `--json` output includes an `evidencePrefill` object for the automated evidence path, automated evidence timestamp, `environment.extensionSource`, and `environment.cleanChromeProfile`. If you pass `-- --profile-dir=...`, the directory must be new or empty. The helper opens `chrome://extensions/` so the unpacked extension id can be copied into `environment.unpackedExtensionId`; pass `-- --launch` to start Chrome directly, or set `SAYLESS_CHROME` if Chrome is not in a standard location.

2. Fill in real test results from `docs/RELEASE_QA.md`.

3. Verify the evidence:

```sh
npm run qa:release:manual
```

The verifier requires `status: "passed"`, the release version to match `package.json`, `package-lock.json`, and `src/manifest.json`, the referenced automated evidence file to exist at the canonical relative path, `automatedEvidenceGeneratedAt` to match that file's `generatedAt`, `testedAt` to be at or after that automated evidence timestamp, and the automated evidence itself to carry `status: "passed"` with internally consistent run window, total duration, command durations, git branch/commit/dirty-state, and exact worktree fingerprint. Automated evidence must include each required release QA command exactly once with `status: "passed"` including `test:e2e:offline-transcription-smoke`, the expected executed command string, and a non-negative duration; unexpected or duplicate command records are rejected. The current `build/manifest.json` version, canonical relative `build/` path, `build/` fingerprint, build byte count, build formatted size, canonical relative bundled-Whisper path, bundled-Whisper fingerprint, bundled-Whisper byte count, and bundled-Whisper formatted size must match the automated evidence. Manual evidence must also confirm a clean Chrome profile, record the 32-character unpacked extension id, and point `environment.extensionSource` at the same canonical relative `build` path fingerprinted by automated QA. Package and CWS evidence preserve and verify the same automated git/worktree provenance so published ZIPs remain tied to the source state that produced the automated QA pass. It also rejects unedited template or fixture placeholder text. Finally, it requires at least two real recordings with unique ids that cover MP4 and WebM, tab/browser/region and desktop/screen/window capture/recording sources, at least one 180-second or longer recording, at least two speaker profiles, and recording notes that describe observed, confirmed, verified, recorded, inspected, tested, opened, or reviewed local/offline recording behavior. It requires inspected exports for MP4, WebM, GIF, WAV, M4A, WebVTT, transcript JSON, and `.sayless-project.json` tied back to listed recording ids with unique filenames matching the recorded export format and notes that describe how each export was opened, played, imported, decoded, previewed, viewed, loaded, listened to, or inspected, verified caption burn-in, export cancel/retry tied to a listed real recording and video format, reveal action evidence that mentions a completed export/download and the Chrome download id, reveal, open, or show-in-folder observation, Save to file evidence that describes the user-chosen file/folder flow or explains why File System Access was unavailable, save-dialog cancellation evidence, structured offline-transcription evidence for bundled-model-ready UI status, a concrete disabled/blocked/offline network isolation method, a failed external HTTP(S) network probe URL/error from the same Chrome profile, a probe result that says the bundled/local Whisper model stayed ready or loaded, transcript quality notes that mention real-speaker/voice/recording quality with word timing, timestamps, accuracy, or usability, cache/regenerate/delete with network disabled, structured silence-suggestion evidence across real codecs/noise with at least two observed suggested quiet ranges and one observed noisy range that was not suggested, structured zoom evidence covering click metadata, keep/remove, preview, persisted keyframes after reopen, and inspected MP4 export framing, structured local-library recovery evidence with duplicate-reopen, sidecar-import, bulk-export-delete, orphan-cleanup, and missing-media-repair operation observations, reviewed release notes, screenshots, `docs/STORE_LISTING.md`, and final store text with searched paid/subscription, premium/trial/entitlement/license/upgrade, plan/membership/locked-feature, locked-behind/pay-to-unlock/upgrade-required gates, account-tier/license-key/activation/contact-sales gates, account/sign-in, and cloud/remote terms, and a `store-text` reviewed artifact name that includes `docs/STORE_LISTING.md`. Publication notes must describe reviewing paid/account/cloud/remote/local-only claims, residual-risk fields must explicitly say no residual risk/no remaining risk or describe the residual risk, and structured checklist evidence must pass for fresh install, recording recovery, timeline persistence, and final no-paid/no-cloud surface checks. Checklist notes and observations must describe observed, confirmed, verified, recorded, inspected, tested, opened, or reviewed local/offline release behavior, and checklist artifacts must identify screenshots, reports, logs, recordings, exports, transcripts, videos, images, JSON/VTT/MP4/WebM/GIF/WAV/M4A files, or notes. Multi-recording evidence requirements must reference distinct listed recording ids; repeating the same id does not satisfy two-recording coverage. Real-recording checklist evidence must identify the listed recording ids used for that check.
